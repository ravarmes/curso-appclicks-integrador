Estudo de Caso 2 — Visão e Requisitos

Objetivo da Semana

Transformar a ideia central do aplicativo em uma Frase de Visão bem definida, escrever as Histórias de Usuário (User Stories) e traduzi-las numa lista clara de Requisitos Funcionais (RF) e Requisitos Não Funcionais (RNF).

Passo a Passo da Atividade

1
Construa a Frase de Visão do Sistema Word / Docs

Reúna as informações da Semana 1 no seguinte template: "Para [Público-Alvo], que [Dificuldade/Problema], o [Nome do App] é um aplicativo que [Principal Solução]. Diferentemente de [Concorrentes], nosso produto oferece [Diferencial]."

2
Escreva os Requisitos Funcionais (Mínimo 8 RFs) ChatGPT / Claude

Peça ajuda a uma IA para listar os requisitos técnicos iniciados por "O sistema deve...". Lembre-se, use o prompt ensinado e inclua as prioridades (Alta, Média, Baixa).

3
Traduza para User Stories (Mínimo 3)

Demonstre que você entende o usuário transformando alguns desses requisitos em User Stories no formato: "Como [Persona], quero [Ação], para [Benefício]."

4
Liste os Requisitos Não Funcionais (Mínimo 4)

Especifiquem sobre como o app deve se portar em questões de Desempenho, Segurança, Usabilidade, etc.

📄
Consolidação no Template Oficial Google Docs

Para concluir as atividades desta semana, reúna todas as informações e artefatos gerados nos passos anteriores e preencha o template oficial de entrega. Faça uma cópia do documento para a sua conta Google e compartilhe-o com o professor.

📱 Estudo de Caso: FinanCampus

Veja como o grupo do FinanCampus definiu a visão e os requisitos do app.

Frase de Visão: Para estudantes universitários que perdem o controle dos gastos mensais, o FinanCampus é um aplicativo mobile que registra e categoriza transações financeiras de forma simples e visual. Diferentemente de planilhas e apps bancários complexos, nosso produto oferece visão rápida do saldo com alertas de orçamento.

Requisitos Funcionais (RF) — Alta Prioridade:

  • RF01 — O sistema deve permitir cadastro e login por e-mail e senha.
  • RF02 — O sistema deve permitir registrar uma transação (tipo, valor, categoria, data, descrição).
  • RF03 — O sistema deve listar todas as transações do mês atual do usuário logado.
  • RF04 — O sistema deve exibir o saldo do mês (rendas − despesas).
  • RF05 — O sistema deve exibir um gráfico de transações por categoria.

Requisitos Funcionais (RF) — Média/Baixa Prioridade:

  • RF06 — O sistema deve permitir definir orçamento mensal por categoria.
  • RF07 — O sistema deve enviar notificação ao atingir 80% do orçamento.
  • RF08 — O sistema deve permitir exportar extrato mensal em PDF.

Requisitos Não Funcionais (RNF):

  • RNF01 — Desempenho: tela principal deve carregar em menos de 2 segundos.
  • RNF02 — Segurança: senhas gerenciadas pelo Firebase Authentication.
  • RNF03 — Compatibilidade: Android 8.0 ou superior.
  • RNF04 — Usabilidade: novo usuário registra primeira transação em menos de 1 minuto.
